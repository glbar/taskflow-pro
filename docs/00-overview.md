# 00. Overview

> **TaskFlow Pro** — 팀 업무 관리 풀스택 웹 앱

---

## 문서 구성 및 역할

| 파일 | 역할 | 이 파일에서 알 수 있는 것 |
|------|------|--------------------------|
| `00-overview.md` | 프로젝트 지도 | 전체 문서 구조, 읽는 순서, 분리 원칙 |
| `01-product.md` | 제품 정의 | 타깃 사용자, 핵심 가치, 주요 기능 목록 |
| `02-specs.md` | 기술 명세 | 기술 스택, 시스템 아키텍처, API·DB 설계 |
| `03-design.md` | UI/UX 설계 | 화면 구성, 컴포넌트 구조, 디자인 시스템 |
| `04-tasks.md` | 작업 계획 | 마일스톤, 스프린트, 현재 진행 상태 |
| `05-conventions.md` | 개발 규약 | 코딩 스타일, Git 브랜치 전략, 커밋 규칙 |

---

## 읽는 순서

```
00-overview → 01-product → 02-specs → 03-design → 04-tasks → 05-conventions
```

- **00**: 전체 맥락 파악 (지금 이 파일)
- **01**: "무엇을 만드는가" — 제품 목적과 기능 범위 확정
- **02**: "어떻게 만드는가" — 기술적 결정과 제약 조건 확인
- **03**: "어떻게 보이는가" — UI 구조와 컴포넌트 설계 파악
- **04**: "지금 무엇을 하는가" — 현재 작업 맥락과 우선순위 파악
- **05**: "어떤 규칙을 따르는가" — 코드 작성 전 반드시 숙지

---

## 왜 이렇게 나누는가 — 관심사 분리(Separation of Concerns)

각 파일은 **하나의 질문에만 답한다.**

| 관심사 | 담당 파일 | 이유 |
|--------|----------|------|
| What (무엇) | `01-product` | 기획 변경이 기술 문서를 오염시키지 않도록 |
| How (어떻게) | `02-specs` | 기술 결정이 기획 의도와 분리되어 독립적으로 관리 |
| Look (모양) | `03-design` | 디자인 변경이 스펙 문서에 영향을 주지 않도록 |
| When (언제) | `04-tasks` | 일정·진행 상태는 수시로 바뀌므로 별도 관리 |
| Rule (규칙) | `05-conventions` | 팀 전체가 참조하는 불변 규약은 독립 유지 |

> 문서가 섞이면 수정 범위를 예측할 수 없다.
> 파일 하나를 열면 필요한 정보만 있어야 한다.
